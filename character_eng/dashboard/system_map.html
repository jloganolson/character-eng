It is meant to answer two questions at once: what kind of experience are we trying to create? and what exactly is happening under the hood when a turn succeeds or fails?

Creative Design

What This Should Feel Like

The system is not aiming for broad assistant competence. It is aiming for a compact dramatic interaction: a character in a world, reacting to a passerby, with visible grounding and enough surprise to feel authored rather than random.

  • Replies should usually be one or two sentences, with a clean stop.
  • The line should feel shaped by the visible scene first, lore second.
  • Oddness is welcome; incoherence is not.
  • The user should feel invited forward, not talked at.
  • When the scene changes, the character should notice quickly.
Technical Contract

What The Runtime Must Guarantee

The stack should remain inspectable even when model output gets messy. That means input/output plumbing, event capture, and runtime controls have to stay reliable enough that a bad turn can be traced to a specific stage instead of turning into folklore.

  • Text, audio, and vision inputs must all land in the same world model.
  • Speech should start quickly enough to feel responsive, with filler only as a cover for latency.
  • All important state changes should be visible in logs or the dashboard timeline.
  • Generated QA runs should be reproducible and reviewable after the fact.
System Topology

How The Big Pieces Fit Together

The simplest mental model is: one main conversation runtime, one optional vision service process, and one optional dashboard surface. The dashboard is for visibility and control. The character loop remains the center of gravity.

Input Layer

Console text, local mic, bridge mic, live camera, injected JPEGs, simulation events, and QA transcripts all become candidate user or environment input.

State Layer

__main__.py maintains the live session. world.py owns facts, people, events, and reconciliation of world changes into durable state.

Model Layer

Chat, thought, expression, director, script checks, reconcile, and automation evaluators are all narrow model calls with different jobs. They should act like small specialists, not one giant brain.

Output Layer

Text streams to the UI, PCM streams to the speaker, dashboard events build a timeline, and QA runs emit JSON plus HTML artifacts that can be annotated.

Core Principle

Keep The Character Loop Legible

  • The fastest route to a fix is isolating whether the problem started in perception, world state, planning, speech timing, or review tooling.
  • Runtime toggles are meant to remove noise without changing code.
  • If a bug cannot be observed in the dashboard or report, the tooling is still incomplete.
Rule of thumb:

If the visible line is bad but the world state is right, the bug is usually prompt or routing. If the visible line is bad because the world is wrong, the bug is usually vision, reconcile, or scenario state.

Operating Modes

Three Main Ways To Run The System

1. Full Local

The main truth path. Local mic and speaker, optional local vision service, standard dashboard.

  • Best mode for feeling real latency, barge-in, and reply shape.
  • Most important mode to keep clean before any remote work.
  • Use this when judging if the character actually works.

2. Browser / Remote

Bridge mode moves audio and camera capture to the browser while keeping the server-side logic largely the same.

  • Useful for remote machines, tunnels, and containerized runs.
  • Less important than full local for truth-testing the creative loop.
  • Dashboard becomes both control panel and transport shell.

3. Full-Stack QA

Automation path that can generate or inject images, transcribe speech, run the character, synthesize replies, and score the result.

  • Best mode for repeatable regressions and artifact-driven review.
  • Produces HTML reports with inline notes saved to logs/annotated/.
  • Ideal for catching drift after prompt or logic changes.

Turn Lifecycle

What Happens During One Good Turn

This is the path to hold in your head while debugging. Almost every broken conversation can be located by asking which numbered stage drifted first.

1

Input arrives

User speech, text, a simulator cue, or a perception event enters the loop. Commands can short-circuit and change runtime state immediately.

2

STT or raw text becomes a message

If audio is used, Deepgram produces transcript text and barge-in signals. That transcript is normalized into the same path used by console or dashboard text.

3

World and scenario context update

Visible events, tracked people, stage progression, and prior facts all shape what the character should think is true before it speaks.

4

Prompt assembly happens

Global rules, character prompt, scenario materials, world summary, and optional system nudges are assembled into the final chat call.

5

Visible reply streams

The chat model generates the response. Expression tags and speech generation begin alongside it. If first audio is slow, short Pocket-TTS fillers can buy time.

6

Post-turn helpers run

Thought, director, script checks, and reconcile adjust world state and prepare the next beat. These should refine the interaction, not hijack it.

7

Events get published

The dashboard timeline, JSON logs, HTML reports, and QA metrics are updated so the turn remains inspectable later.

8

Idle logic may schedule the next beat

If auto-beat is on, the loop can schedule a follow-up beat after speech finishes. If it is off, the user fully controls pacing.

Processes & Threads

Where Concurrency Actually Lives

The system has several overlapping loops, but they are not all equal. The key discipline is to treat only a few of them as authority-bearing and the rest as feeder or helper paths.

Main character process authoritative session state

  • Runs the main conversation loop in character_eng/__main__.py.
  • Owns scenario stage, chat history, world state application, command handling, and dashboard emission.
  • Should be the only place where a user turn is finally interpreted into character action.

Voice helper threads latency-sensitive I/O

  • Mic feeder thread sends audio to Deepgram.
  • Deepgram listener thread turns partials/finals into transcript and barge-in events.
  • TTS generation/receive path streams PCM to the speaker queue.
  • Filler timer can briefly emit Pocket-TTS clips while waiting for primary speech.

Vision service process separate HTTP service

  • Runs in services/vision/app.py.
  • Can be backed by a live camera, browser frames, replay files, or generated images.
  • Exposes health, snapshot, injection, and question/target endpoints consumed by VisionManager.

Dashboard / bridge surface observability and control

  • Displays event history and current runtime controls.
  • Can inject text commands and show reviewable state.
  • Should never become the hidden source of truth for the character.
Vision Design

How Visual Grounding Is Supposed To Work

Frame source

A frame comes from a local camera, the browser bridge, a replay file, or generated imagery from the QA harness.

Vision service

The service tracks people and scene targets, answers configured VLM questions, and can use SAM targets to focus on likely objects that matter for the beat.

Snapshot distillation

VisionManager polls snapshots and converts raw service output into compact perception events usable by the character loop.

World state merge

Perception enters the world and reconcile path as claims about the current scene. This is where false positives, missing targets, or stale observations can poison the turn.

Creative consequence

The character should then speak as if it actually noticed the object or person. If the VLM saw a bottle but the line ignores it, the integration is weak even if the vision service was technically correct.

Good vision behavior

  • Notices the few objects that matter for the current beat.
  • Updates quickly when the user enters, leaves, or reveals something.
  • Produces facts that can survive into later turns without getting weird.

Common weak points

  • Noisy or stale snapshots produce old facts.
  • SAM targets are too generic or missing key props.
  • Scene interpretation is correct in logs but never affects the final line.

What to tune first

  • Question set for the current beat.
  • Target list for likely props or gestures.
  • How perception is summarized into the chat context.
Prompt Stack

Which Prompts Exist And Why

The stack works better when prompts are narrow and explicit. The visible reply should be controlled by a small set of strong constraints, while support prompts clean up state or stage management around it.

Global rules

Sets the broad behavioral frame: short turns, no AI self-reference, and constraints that should apply across characters.

Character prompt

Defines voice, tone, quirks, and what makes Greg feel specific instead of generic.

Scenario materials

Supply beat-specific framing, stage objectives, and a sense of where the conversation is currently supposed to be.

World summary

Injects visible scene facts, people, and recent events so the line can be grounded in the moment.

System nudges

Short-turn automation or runtime hints can be appended before submit to bias the immediate line without rewriting the whole stack.

Support prompts

Thought, director, reconcile, script checks, and evaluators are support systems. They should sharpen the scene, not dominate it.

Runtime Controls

How To Iterate Without Restarting The Whole App

The dashboard and CLI now expose the same control surface via /threads. These controls mute subsystems logically instead of tearing down the process graph.

Reconcile

Stops new reconcile jobs from starting. Use it when background fact cleanup is making it hard to judge the immediate conversational line.

Vision

Mutes vision event ingestion into the main loop while keeping the service alive for snapshots and diagnosis.

Auto-beat

Disables the idle follow-up timer. Useful when you need fully user-driven pacing during testing.

Filler

Toggles short Pocket-TTS latency fillers like "oh", "hmm", and "okay" while waiting on primary speech.

Status: /threads status
Disable a subsystem: /threads reconcile off, /threads vision off, /threads auto-beat off, /threads filler off
Iteration Workflow

How To Review And Feed Fixes Back In

Live dashboard

Use it to watch events, current state, controls, and the rough timing of a session while the app is running.

HTML QA reports

Use the full-stack harness when you want a stable artifact with images, turns, timings, and evaluator output.

Inline annotations

Open a report with uv run -m character_eng.open_report <report.html>, annotate individual turns, and save structured notes to logs/annotated/.

Agent feedback loop

Those notes are the cleanest way to turn human taste into concrete iteration requests like "too long", "great open-ended question", or "vision missed the bottle".

Failure Signatures

What Breakage Usually Looks Like

Reply is too long

The short-turn prompt is losing authority, or support prompts are bloating context until the visible line stops feeling crisp.

Reply ignores obvious visible objects

Either the vision service did not notice the thing, the summary dropped it, or the chat prompt is not weighting visible facts strongly enough.

Audio feels late or dead

Look at TTS startup timing, speaker queue behavior, and whether fillers are masking or merely hiding a deeper speech latency issue.

World facts get weirdly polluted

Reconcile output or event carryover is introducing stale or malformed facts. This usually shows up as repeated irrelevant observations or awkwardly prefixed fact text.

Internal artifacts drift into nonsense

The visible reply may still look grounded while planner, director, or script artifacts start talking about things like coupons or hidden lore. That means the support layer is no longer serving the scene cleanly.

Where To Edit

If You Want To Change Behavior, Start Here

Conversation loop

character_eng/__main__.py

  • Main runtime flow
  • Command handling
  • Runtime controls
  • Scenario/stage transitions

World and reconcile

character_eng/world.py

  • Fact storage and people state
  • World updates
  • Reconcile sanitization

Voice path

character_eng/voice.py and character_eng/filler_audio.py

  • STT wiring
  • Speaker buffering
  • TTS startup
  • Latency filler behavior

Vision service

services/vision/app.py and character_eng/vision.py

  • Snapshot contract
  • Tracking and questions
  • Perception summarization

Prompts and character

prompts/

  • Global behavior constraints
  • Character voice
  • Scenario framing
  • Automation short-turn bias

QA harness and reports

character_eng/qa_full_stack.py

  • Generated scenes
  • Automation runs
  • HTML report rendering
  • Annotation export
Recommended debugging order:

First confirm the world is right. Then confirm the visible reply is short and grounded. Then tune support prompts, filler, and automation. Do not start by changing ten prompts at once.
